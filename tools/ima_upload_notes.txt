================================================================
ima_upload.bat  使用说明 / 版本记录
（本说明独立存放：.bat 文件内禁止出现任何中文字符——
 既不能是 REM/:: 注释，也不能是 echo 输出。原因与排错见下方
 【常见故障：拖放 bat 闪退】一节）
================================================================

【用途】
一键把 Markdown 文件导入 IMA 知识库（走 ima_upload.py 官方 OpenAPI）。
支持两种触发方式：
  1) 双击（不拖放任何东西）：导入 ima_config.ini 的 SRC 来源文件夹里的全部 .md
  2) 拖放文件 / 文件夹到 bat 上：导入被拖放项里的 .md
     - 拖放的是文件：是 .md 就导入，非 .md 自动跳过
     - 拖放的是文件夹：导入该文件夹内的全部 .md（仅一层，不递归子目录）

【使用前的一次性配置】
配置已外置到独立 ini，bat 本身不再写任何配置：
  1) 把同目录的 ima_config_sample.ini 复制为 ima_config.ini
  2) 用记事本打开 ima_config.ini，填写下面三项（以 ; 开头的行是注释，可忽略；
     等号前后可加空格、值里不要加引号）：

       KB_ID = 你的知识库ID          ; 必填。ima.qq.com 知识库设置里查看
       SRC   = C:\你的\导出目录      ; 双击模式用。留空则双击时必须拖放
       PY    = C:\路径\python.exe    ; 可选；python 不在 PATH 时才需要

说明：
  - ima_config.ini 已加入仓库 .gitignore，不会随提交泄露你的知识库 ID / 本地路径；
    仓库只保留 ima_config_sample.ini 模板。
  - KB_ID 为空时双击/拖放都会先报错提示，不会误传。
  - SRC 只取一层目录里的 .md；如需递归或别的规则，改 bat 或自行调用 ima_upload.py。

【凭证（IMA 账号）】
bat 只是调用 ima_upload.py，凭证由它读取，提前准备好其一即可（配置里不存凭证）：
  A) 环境变量 IMA_CLIENT_ID / IMA_API_KEY
     （临时：set IMA_CLIENT_ID=... ；永久：setx IMA_CLIENT_ID ... 后重开窗口）
  B) 文件 %USERPROFILE%\.config\ima\IMA_CLIENT_ID 与 IMA_API_KEY
导入前脚本会自动校验该 KB_ID 是否当前账号可写，无权限会立即报错并列出可写库。

【Python（PY 即 Python 可执行文件）】
PY 就是用来跑 ima_upload.py 的那个 Python（命令名或完整路径）。
  - 默认自动探测：先试 python，找不到再试 py 启动器。
  - 若你的 Python 是 Miniconda 且未注册为默认 python（PATH 里没有 python），
    在 ima_config.ini 直接填 PY 的完整路径即可，例如：
        PY = C:\Users\你的用户名\miniconda3\python.exe
    或某个虚拟环境：
        PY = C:\Users\你的用户名\miniconda3\envs\myenv\python.exe
  - 填好后会做一次可用性校验（python --help 能跑通才算数）；跑不通会明确提示
    检查该 python 是否可用、依赖是否安装（pip install -r tools/requirements.txt）、
    或把 PY 改成正确的 python.exe 路径。
依赖（与 ima_upload.py 相同）：见同目录 requirements.txt
（一次性安装：pip install -r tools/requirements.txt；
 HTTP 桥模式 --serve 不需要 watchdog，仅目录监视模式必需）

【入库标题规则（简版，详见 README）】
不显式指定标题时，脚本从文件名自动提取真实标题并去掉 Chat2Knowledge 前缀；
IMA 要求 title 与 file_name 完全一致（含 .md），最终统一写成「标题.md」。

【常见故障：拖放 bat 闪退（窗口一闪就关）】
现象：把 .md 文件拖到 ima_upload.bat 上，黑窗口一闪即关、什么都没发生。
bat 闪退通常是「语法错误」导致 cmd 立即中止。按致命程度排列，根因基本都在这几类：
  (1)【最致命，本次真正根因】echo 文本里出现未转义的括号 ( )，而且该行又处在
      if (...) ( ... ) / for (...) ( ... ) 这类带括号的代码块内。cmd 会把 echo
      里的 ( 当成「新代码块开始」、) 当成「块结束」，导致括号不匹配 → 语法错误
      → 整段脚本中止 → 闪退。（本次出问题的正是 echo 行里写了 "(e.g. ...)" 与
      "(s)"。）修复：echo 文本里不要出现裸括号，或改用 ^( ^) 转义。
  (2) bat 文件是 LF（Unix 换行）而非 CRLF（Windows 换行）。cmd.exe 要求 CRLF，
      LF 会让多行块 / 标签解析失败 → 语法错误 → 闪退。本项目 bat 由脚本生成时
      须写出 CRLF；若你用编辑器另存 / 改过，务必选 CRLF（VS Code 右下角切换；
      记事本默认就是 CRLF，一般无此问题）。
  (3) bat 里残留了中文字符（echo 或注释）。中文 Windows 用 GBK 解析 .bat 的
      UTF-8 字节，某些中文字节会被误读成 ) & > 等命令元字符 → 语法错误 → 闪退。
      所以本 bat 保持纯英文 ASCII，中文说明集中放在本 notes.txt。
  (4)【最隐蔽】子程序里出现裸命令连接符 &（如 `if ... set ... & goto label`），
      且这个子程序是从某个 (...) 括号内代码块里 call 出来的。cmd 在「块内 call
      子程序、子程序里又有 & 复合命令」这种嵌套下会解析错乱，报
      "& was unexpected at this time." 而立刻中止。本次真正的致命点正是
      :trim 子程序（在 for /f (...) do (... 配置解析块内被 call :trim 调用）
      里的两行 `set ... & goto trim_loop`。修复：彻底删除该子程序，改用一个
      嵌套 for /f "eol= tokens=*" 直接给配置值去首尾空格（for /f 自带去空白），
      不再需要子程序，也就没有 & 与 EnableDelayedExpansion。
自检：用记事本打开 ima_upload.bat，（a）搜不到任何中文；（b）VS Code 看右下角
  为 CRLF。bat 现已不在磁盘写任何日志文件，所有信息都直接显示在窗口里；若仍
  闪退，请从 cmd 进入 tools 目录后手动运行 `ima_upload.bat "某个.md"`，窗口不
  会一闪即关，能直接看到 cmd 报的具体错误（如语法错误），把该报错发我即可定位。

【常见故障：KB_ID not set（明明配了 ima_config.ini 却报没设）】
现象：已复制 ima_config_sample.ini 为 ima_config.ini 并填好 KB_ID，bat 仍报
  [error] KB_ID not set. Copy ima_config_sample.ini to ima_config.ini and set KB_ID=...
根因：ima_config.ini 里写成 `KB_ID = xxx`（等号前面多了个空格）。bat 用 for /f 按
  `=` 切分，切出的 key 是 "KB_ID "（带尾随空格），和代码里比对的 "KB_ID" 不相等
  → 永远设不上。value 前面的空格会被内层 for /f 去掉，但 key 名后面的空格没处理。
  （v1.1.5 已修复：给 key 也加一层去空格的 for /f，等号前后有 / 无空格现在都能读。）
自查 / 临时绕过：
  - 直接把 ima_config.ini 改成 `KB_ID=xxx`（等号前后都不留空格）即可立刻生效；
  - 或升级到 v1.1.5 的 ima_upload.bat，两种写法都支持。
  - 若仍报 not set：确认 ima_config.ini 与 bat 在同一文件夹（bat 会提示
    "ima_config.ini not found" 而非 "KB_ID not set"）；有 found 提示但 KB_ID 仍空，
    说明那一行的 key 没匹配上（检查是否写成了注释、是否等号两边有奇怪字符、
    文件是否被存成了带 BOM 的 UTF-8 导致首行被加料）。
  注：KB_ID 形如 base64 长串、结尾常带 =，这是正常的，bat 会原样读取并传给脚本。

【成功 / 失败提示（用户最关心）】
bat 直接把 Python 的真实输出显示在屏幕上（不再写任何磁盘日志文件），成功失败都看得见。
显示内容：
  - 每个 .md 上传时，Python 的真实输出（成功打 [OK] 已导入知识库 / 失败打 [ERR] ...）
    直接显示在窗口。
  - 紧接着打明确的横幅：`[OK] uploaded: <路径>` 或 `[FAIL] upload failed, exit code N: <路径>`。
  - 所有文件处理完后打汇总：`[summary] uploaded OK : N` / `[summary] failed : M`
    （都没上传则提示 no files were uploaded）。
  - 设 `PYTHONIOENCODING=utf-8` + 顶部 `chcp 65001`，保证 Python 输出与屏幕的中文不乱码。
实现细节：python 直接输出到屏幕，紧接 `set "RC=%errorlevel%"` 捕获退出码，按 RC 打
  OK/FAIL 横幅并 `set /a OK_CNT/FAIL_CNT+=1`（v1.1.8 起不再使用临时文件 _upload_out.tmp
  与 ima_upload.log，信息全部实时上屏）。

【常见故障：上传成功却在收尾时报 `: was unexpected at this time.` 并闪退】
现象：Python 输出 `[OK] 已导入知识库: ...` 正常打完，紧接着一行
  `: was unexpected at this time.` 然后窗口一闪即关。
根因：bat 的 `:upload` 子程序里那段「if %RC%==0 ( ... ) else ( ... )」收尾块，在
  **执行前**就会被 cmd 一次性解析。即便本次上传成功、根本没走 else 分支，else 块里
  那行 echo 文本 `… (exit=%RC%): …` 中的字面圆括号，会被 cmd 当成「新代码块开始/结束」，
  其后的 `: %ARG%` 就成了非预期的 `:` → 报 `: was unexpected at this time.`。
  也就是说：**只要 (...) 块内的 echo 文本里出现字面圆括号，整个块在解析阶段就崩，
  与成败、是否真执行到那一行无关**。
修复（v1.1.7）：把那行 echo 的圆括号去掉，改为 `echo [FAIL] upload failed, exit code %RC%: %ARG%`
  （冒号 mid-text 是安全的，问题只出在 `(`）。并全文扫描确认：再无任何 echo 行文本
  内含字面圆括号。
自查同类隐患：凡是 `if (...) ( echo ... )` / `for (...) do ( echo ... )` 这类块内的
  echo，文本里一律不要出现字面 `(` `)`；要表达「（…）」请用中文全角括号 「（）」，
  或直接写成 `exit code xxx` 这种不含半角括号的写法。

v1.1.7（2026-07-16）
  - 修复「上传成功、收尾时报 : was unexpected at this time. 并闪退」：根因是
    :upload 子程序里 `if %RC%==0 ( ... ) else ( ... )` 收尾块在**执行前**就被 cmd
    一次性解析；else 分支 echo 文本 `… (exit=%RC%): …` 里的字面圆括号被 cmd 当成
    新代码块开/闭，其后 `: %ARG%` 变成非预期的 `:` → `: was unexpected`。此块无论
    成败、是否真的执行到 else 都会崩（解析阶段即报错）。修复：去掉该 echo 的圆括号，
    改为 `echo [FAIL] upload failed, exit code %RC%: %ARG%`，并全文扫描确认再无 echo
    行文本含字面圆括号。字节审计：无 BOM / 135 CRLF、0 裸 LF / 0 非 ASCII / 0 `!` /
    0 裸 `&` / 括号 24/24 平衡。
  - 教训：bat 里「(...) 块内的 echo 文本出现半角圆括号」会令整个块解析失败（哪怕那
    一行从不执行）；块内 echo 一律用中文全角括号「（）」或不含半角括号的写法。

v1.1.6（2026-07-16）
  - 新增「上传成功 / 失败的可视化提示」：之前 Python 输出全进日志、屏幕只打
    `[upload] 路径` 一行，用户看不到结果。现把 Python 真实输出（成功 [OK] 已导入
    知识库 / 失败 [ERR] ...）同时显示在屏幕并写日志；每个文件追加 `[OK] uploaded:`
    或 `[FAIL] upload failed, exit code N:` 横幅；结束打 `[summary] uploaded OK : N` /
    `[summary] failed : M` 汇总。设 PYTHONIOENCODING=utf-8 + 顶部 chcp 65001 防中文
    乱码；临时输出走 _upload_out.tmp，展示并记日志后于结束前删除。实现：python 输出
    重定向到临时文件 `> "%TMPFILE%" 2>&1`，随即 `set "RC=%errorlevel%"` 捕获退出码，
    再 `type` 临时文件到日志与屏幕，按 RC 打横幅并 `set /a OK_CNT/FAIL_CNT+=1`。

v1.1.5（2026-07-16）
  - 修复「ima_config.ini 已配好 KB_ID 却报 KB_ID not set」：v1.1.4 的解析只给 value
    去空格，没给 key 去空格。当 ima_config.ini 写成 `KB_ID = xxx`（等号前带空格）时，
    for /f 切出的 key 是 "KB_ID "（带尾随空格），与 "KB_ID" 比不相等 → 永远设不上。
    修复：在外层按 = 切分后，再套一层 for /f "tokens=1 delims= " 给 key 去首尾空格
    （for /f 自带去空白），value 仍用内层 for /f "eol= tokens=*" 去前导空格。
    现已能正确读取 `KB_ID = xxx`、`KB_ID=xxx`、`SRC = 路径`、`PY = 路径`
    （含空格、含结尾 = 的 base64 型 KB_ID 均正常）。用脚本以真实 ima_config.ini 模拟
    for /f 切分验证：KB_ID / PY 均正确提取、SRC 留空（拖放模式不需要）。
  - 教训：bat 里 key=value 解析，等号两侧的空格都要处理；只 trim 一侧会漏掉另一侧。

v1.1.4（2026-07-16）
  - 修复「用户从 cmd 直跑 bat 报 & was unexpected at this time.」：真正的致命点
    是 :trim 子程序里的两行 `if ... set ... & goto trim_loop` —— 该子程序从
    for /f (...) do (...) 配置解析块内被 call 出来，cmd 在「块内 call 子程序 +
    子程序内含 & 复合命令」这种嵌套下解析错乱，报 & 错误并中止。这是继 v1.1.3
    （echo 裸括号）之后、压在下面的第二层解析错误，所以 v1.1.3 修完仍崩。
  - 修复方式：彻底删除 :trim 子程序与其 call，配置解析改为「外层 for /f 按 =
    切分 key/value，内层 for /f "eol= tokens=*" 给 value 去首尾空格」（for /f
    自带去空白），并去掉 EnableDelayedExpansion（已无 !var! 需求）。现全文零 `!`、
    零裸 `&`（仅剩 2>&1 重定向）、括号 20/20 平衡、CRLF、纯 ASCII、无 BOM。
  - 教训：bat 里「从 (...) 块内 call 一个含 & 的子程序」是经典雷区；配置/文本
    处理优先用纯 for /f 嵌套而非 call 子程序 + &。
